会议背景：
本次会议围绕“办公资料整理与行动项提取助手”的最小可运行 Demo 进行需求评审。该 Demo 需要在面试展示中体现 Agent Loop、工具调用、结构化输出以及一个明确的产品亮点。会议目标是明确第一版功能范围、技术实现路径、交付物和后续风险。

一、产品目标讨论
李明说明，第一版不要做得过大，重点不是覆盖所有办公场景，而是把一个典型场景做闭环。建议优先选择“会议记录整理”作为入口，因为输入简单、输出明确，也容易展示 Agent 的多步执行过程。最终产物应包括会议概要、关键结论、行动项、风险点和 Agent 执行过程摘要。

赵敏补充，Demo 必须避免看起来像一次性问答。Agent 至少要有读取文件、分析任务、提取行动项、生成报告、写入文件几个步骤，并且每一步都要记录 thought、action、observation。这样在最终报告里可以展示 Agent 的执行路径。

二、功能范围确认
团队讨论后确认第一版 MVP 包含以下能力：
1. 读取本地 txt 或 md 格式的会议记录文件；
2. 从会议记录中提取会议主题、关键结论、行动项、负责人、截止时间和风险点；
3. 对每个行动项保留原文依据片段，便于用户追溯；
4. 生成 Markdown 格式的会议整理报告；
5. 在报告末尾展示 Agent 的执行过程摘要；
6. 提供 demo/run_demo.py，方便面试官一键运行。

暂不纳入第一版的内容包括：PDF 解析、Word 解析、音频转写、多人协同编辑、Web UI 和数据库存储。李明强调，先保证本地命令行 Demo 稳定运行，再考虑扩展。

三、技术方案讨论
王磊建议先完成工具层，包括 FileReader 和 FileWriter 两个基础工具。FileReader 负责读取 demo/input/meeting_notes.txt，FileWriter 负责把最终报告写入 demo/output/meeting_report.md。工具返回值需要统一格式，包含 success、output 和 error，方便 Agent 判断下一步。

赵敏认为 Agent Loop 可以先做简化版，不一定一开始就完全依赖模型动态决策。第一阶段可以采用固定流程：读取文件 -> 抽取信息 -> 构建报告 -> 写入文件；第二阶段再加入 planner.py 和 executor.py，让 Agent 根据任务生成计划并选择工具执行。

陈雪提出，虽然第一版没有 Web UI，但输出报告应尽量易读，表格结构要清晰。行动项表格建议包含“行动项、负责人、截止时间、优先级、依据片段”五列。若会议记录没有明确截止时间，应标记为“待确认”，不要编造。

四、产品亮点讨论
团队一致认为第一版的产品亮点定为“可追溯的行动项提取”。林佳指出，很多会议总结工具只给结果，但用户无法判断结果是否来自原文。我们的报告应在每个行动项后附带依据片段，这样用户可以快速检查 Agent 是否误解了原文。

李明补充，这个亮点也适合面试展示，因为它既体现用户价值，也体现技术判断：Agent 不是只生成漂亮文本，而是要输出可验证、可追溯的信息。

六、风险与问题
周航提醒，会议记录中的负责人和截止时间可能并不总是明确出现。如果 Agent 强行补全，可能会产生幻觉。团队决定：如果原文没有明确负责人或截止时间，就输出“待确认”，并在风险点中说明。

赵敏指出，如果后续接入模型，可能会受到上下文长度和输出格式稳定性的影响。短期解决方案是使用固定字段和简单校验，要求输出必须包含指定章节；长期可以引入 JSON Schema 或结构化解析。

王磊提到，文件路径在不同操作系统下可能存在兼容问题。当前开发环境是 Windows，但代码中应尽量使用 pathlib.Path，而不是硬编码路径分隔符。

陈雪提出，报告中的依据片段不能太长，否则表格会难以阅读。初步约定每个依据片段控制在 80 字以内，必要时可以截断并保留关键句。

七、会议结论
1. 第一版项目聚焦“会议记录整理与行动项提取”，不扩展到复杂办公全场景。
2. Demo 必须体现 Agent Loop 和工具调用过程。
3. 工具层先实现 FileReader 和 FileWriter，后续再增加 TextExtractor、KeywordSearch 等工具。
4. 产品亮点确定为“可追溯的行动项提取”。
5. 最终输出采用 Markdown 报告，文件路径暂定为 demo/output/meeting_report.md。
6. 若信息缺失，Agent 应标记为“待确认”，不得编造负责人、时间或结论。
